MyBatis 基础知识

发表于2023-02-24|更新于2023-03-15

1. 什么是 MyBatis

MyBatis 是一个 Java 持久层框架，包括 SQL Maps 和 Data Access Object。

可以支持定制化 SQL、存储过程、高级映射，几乎避免了所有的 JDBC 代码和手动参数以及获取结果集。

可以使用 xml 或者注解将接口和 POJO 映射成数据库中的记录。

在 JavaEE 三层架构中，MyBatis 属于持久层框架。

MyBatis 是一种半自动 ORM 框架，需要手动编写 SQL

相对应的 Herbernate 就是一种自动 ORM 框架，自动生成 SQL 语句。

4. MyBatis 的事务管理机制

我们通过 <Transaction> 配置事务管理器，其 type 属性有 JDBC、MANAGED 两个取值。

前者表示采用 JDBC 管理事务，当我们使用 sqlSessionFactory.openSession() 方法获取 SqlSession 时，默认将 autoCommit 设置为 false，这种设置会关闭 JDBC 的自动提交，开启事务管理。如果要开启自动提交，只需要使用 sqlSessionFactory.openSession(true) 即可，此时是事务管理将会被关闭，SQL 会被自动提交，无法回滚。

因为 JDBC 事务管理器默认关闭，即如果不指明 conn.setAutoCommit(false)，JDBC 默认关闭事务，开启自动提交。因此传入 true 时会底层会跳过 conn.setAutoCommit(false)。

后者表示将事务管理器交给其他容器管理。如果没有配置其他管理器，则事务不会被开启，因为根本没人管。

8. 使用 MyBatis 动态代理

MyBatis 提供了代理方法来根据 Dao 接口自动生成代理类，使用动态代理要求 Mapper.xml 文件中的 name space 是对应接口的全限定名，id 是对应的方法名。

SqlSession 类提供了 getMapper 方法，该方法根据传入的 class 类型匹配对应的 mapper.xml，并根据 class 和 mapper.xml 的信息生成代理类。

此处传入的 class 类型是 Dao 层的接口，我们直接调用接口所定义的方法即可，其代理类底层仍然使用了前面所使用的 selectOne 等原始方法。

关于 MyBatis 的底层原理和动态代理的原理，请见 MyBatis 原理一文。

9. #{} 与 ${}

两者都是占位符。

#{} 使用 PreparedStatement：先编译 SQL 语句，然后将参数传入。

${} 使用 Statement：先将参数传入并拼接 SQL 语句，然后编译。

后者存在 SQL 注入的风险。

#{} 传字符串值时，会将参数设置为 'param'，即自动在两边加上单引号，而 ${} 则是直接进行字符串拼接。因此在传递非 SQL 关键字的参数时，使用 #{}，如果传入关键字，例如 asc 等排序关键字，以修改 SQL 的行为，需要使用 ${}。

两者的使用并无绝对规定，对于参数尽量使用 #{} 以避免 SQL 注入，而如果拼接 SQL 时需要不加单引号以实现某些特定行为，则可以使用 ${}。

上述内容比较浅显，如果对 JDBC 和 SQL 较为熟悉，则只需要记住：

#{} 会将使用 preparedStatement 将其所在的位置设置为 ? 占位符，然后根据类型推断调用 setXxx(index, value) 传参。

${} 则是直接将其所在的位置替换为传入的值。

12. MyBatis 的查询返回值

返回一个或多个 POJO 已经讨论过，在 resultType 中均只传入 POJO 类型，而实际使用单个 POJO 或者 List<POJO> 来接受结果。

如果返回多个值但用一个对象接受，会报错。但是单条记录可以用 List 接受。

如果没有合适的 POJO 可以使用 Map<String, Object> 或者 List<Map> 来接受。

还可以使用 Map<Map<String, Object>> 来替代 List<Map> 以求更方便的操作，使用这种方式时要在 Mapper 接口上使用 @MapKey(“”) 注解来指定一个字段作为 key。

在前面我们通过在 SQL 语句中使用关键字 AS 来解决数据库命名和 POJO 字段命名不一致的问题，除了这种方法之外还有两种方法可以实现字段映射。一种是使用 resultMap 标签在 xml 文件中映射，一种是开启驼峰映射。

驼峰自动映射要求 POJO 字段名严格遵循驼峰命名，数据库字段严格遵循下划线命名，且两者只有写法不同，拼写完全相同。

13. MyBatis 动态 SQL

某些业务场景下需要根据用户的选择动态调整 SQL 中的参数或者关键字。

if 标签

if 标签用于进行判断，判断结果为真则将标签内的语句拼接到 SQL 中，该标签的 test 属性为必须的，其中编写判断表达式，该表达式获取参数值的方式类似 #{}，使用 @Param 值或者 POJO 属性名或者 arg0 来获取对应的参数，其逻辑关键字采用 and、or。

需要注意判断为假拼接失败时如何保证 SQL 语句合法的问题，此处通过添加永真条件来避免 SQL 的格式错误。这样操作将导致所有参数为空时查询所有数据。

where 标签

使用 where 标签可以更灵活的操作。该标签负责 where 子句的生成，可以自动的处理一些潜在的格式问题。

where 内部所有的 if 标签如果均不成立，则 where 子句不会生成。

where 标签会自动删除多余的 and 关键字，上述例子中第一个 if 如果成立，则其 and 关键字会被自动删除以避免格式问题。

只能自动去除前面多余的 and 和 or。

trim 标签

trim 标签有四个属性，prefix 和 suffix 设置前后缀，该前后缀将会添加在 trim 标签的之前和之后，prefixOverrides 和 suffixOverrides 将会删除 trim 标签内部的子标签的内容的前后缀。

此处的配置会在 trim 之前添加 where 前缀，并将其内部 if 语句的后缀 and | or 两者去除。

如果 trim 标签内部全部为假，不生成任何内容，则前后缀也不会生成。

set 标签

用于 update 语句中动态配置。

在使用 update 语句时，我们可能只是想要更新某一列的数据，如果每一列单独写一个 update 语句未免过于繁琐，而如果在一个 update 语句完成所有更新，我们又需要将不想更新的列设置为原本的值，而要设置为原本的值我们有需要查询其原本的值，这过于负责。

set 标签为我们提供了一种方法，即是我们在 update 中写了所有列的更新，但如果传入的值为空，set 标签会自动为我们跳过这个列，从而避免将正常数据替换为空值的错误。

set 标签会自动去除后面多余的逗号

choose when otherwise 标签

这三个标签类似于 if-else 语句。

choose 标签只会选择内部一个标签执行，其逻辑类似于 if-else 语句，当一个 when 标签通过时，其他标签将不在生效，otherwise 标签在其他 when 全部失效时执行。

foreach 标签

这是一个循环标签，对于传入的数组或者列表进行遍历操作。

foreach 标签有三个必须的属性，collection 属性指定传入的集合的参数名，item 属性是一个临时变量，存储从集合中取出的条目，separator 用于指定每个条目之间的分隔符。open 和 close 标签是可选的，这里用于指定 in 关键字后面的小括号。

如果传入的是 POJO 的列表，想要取用 POJO 的属性，可以使用 #{item.field} 的形式。

sql 和 include 标签

用于代码复用。

很多代码具有部分重复性，因此可以通过声明 SQL 片段以便复用这些代码。

include 标签所在的地方会被替换为 sql 标签中的内容。

15. MyBatis 缓存

MyBatis 会将 DQL 语句的查询结果缓存到内存中，以提高查询效率。

MyBatis 支持三级缓存：

  1. 一级缓存：MyBatis 提供，数据存储到 SqlSession。

    一级缓存默认开启，对于同一个 SqlSession 执行同一条 SQL 语句时，该 SQL 语句只会在第一次执行时真正执行一次，后续的执行都是直接从缓存中取结果。

    这里指编译后 SQL 语句一样且数据库未更新，即当查询条件改变或者进行了数据库的更新操作时，一级缓存会失效。

    也可以调用 SqlSession 的 clearCache 方法清空一级缓存。

  2. 二级缓存：MyBatis 提供，数据存储到 SqlSessionFactory。

    SqlSessionFactory 提供整个环境的缓存，需要手动配置开启。

    要求使用二级缓存的对象是可序列化的，即 POJO 实现 Serializable 接口。

    由此可见 Mybatis 缓存的是封装好的 POJO。

    SqlSession 对象关闭或者提交后，一级缓存的数据会被写入二级缓存，此时二级缓存才生效。

  3. 第三方集成，EhCache等。

集成第三方缓存是为了代替二级缓存，一级缓存无法被替代，此处不再给出配置示例。

17. MyBatis 分页插件

MySQL 使用 limit 关键字实现分页，可以提供两个参数，即 startIndex、Size。

只提供一个参数，则默认 startIndex 为 0。

MyBatis 要实现分页功能，只需要固定 Size，修改 startIndex，即可达到分页效果。

startIndex = (pageNum - 1) * pageSize，从页码推断 startIndex。

单纯的分页获取信息并不难，只要传入 startIndex 和 pageSize 即可查询，但是其他信息，诸如当前表中数据可以分多少页，下一页操作或上一页操作是否可用等，这一些信息需要多个 SQL 语句和业务方法配合实现。

需要注意

上述代码可能让人摸不着头脑，为什么单纯的将结果集合传递给 PageInfo 就可以直接获取分页信息了呢，难道 PageHelper 根据查询结果重新去数据库中查询其他信息吗？

关键在于查询结果集合，表面上返回了一个 List，其实其本质是一个 Page extends ArrayList，并不是我们常用的集合类，而是 PageHelper 提供的集合，这个集合中就封装了许多和分页有关的信息。PageInfo 实际上是根据这些信息来构建的，而不是自己重新查询。

PageHelper 设置了拦截器，在执行 SQL 语句时自己获取了许多额外的信息，例如会多执行一条语句查询表中的条目总数。将这些信息和查询结果封装在 Page 中，我们将查询结果传递个 PageInfo 时，就根据这些信息构建对象。

如果我们不开启分页，即不调用 startPage 方法，PageHelper 不会对查询进行拦截，返回值也就是正常的 Java 集合，当我们将这个 Java 集合传递给 PageInfo 时，PageInfo 会通过 instanceof 关键字判断其不是 Page 类的实例，由于其中没有封装数据，，从而将 pageNum 设置为 1，然后从集合中获取能得到的信息，例如长度等，来构建一个 PageInfo。在这种情况下 PageInfo 的信息是不可信的，如果我们在 SQL 语句层面手动实现了 limit 分页，PageInfo 是无法感知的，其 pageNum 信息就是错误的，因此如非必要，请不要把其他集合传递给 PageInfo。

18. MyBatis 注解开发

注解开发省略了 Mapper.xml 文件。通过注解直接在接口上直接标明 SQL 语句。

但注解基本只能实现简单的 SQL 语句，因为注解底层也是基于 xml 语法，对于复杂的 SQL 语句，实际上就是在注解中编写 Mapper.xml，其可读性和可维护性很差。

同时也不建议 xml 和注解混用，这同样会增加维护难度，因此基本只使用 xml。

@Select @Update @Delete @Insert @Results

前四个注解不再多做解释，最后一个注解其实就是 ResultMap 的注解形式。

关于注解的使用不再过多讨论，请自行了解。

19. Mybatis 与 Spring 整合

使用 mybaits-spring 包来整合，将 SqlSession 和事务控制交给 Spring 管理。

SqlSessionFactoryBean 由 mybatis-spring 包提供，是一个 SqlSessionFactory 的实现类，可以为其指定数据源和配置文件路径，该 Bean 会在创建时自动加载配置文件从而创建 SqlSessionFactory。

MapperScannerConfigurer 也是由 mybatis-spring 提供，该 bean 会扫描 mapper 接口并自动生成其实现类，然后将实现类注入容器，这样我们就可以使用依赖注入在其他 bean 中使用 mapper 对象。

将事务管理器注入容器，然后使用 AOP 对切点进行事务增强。service 层的方法被增强后就不再需要手动调用事务的提交。

advice 标签中并没有指定 transaction-manager 属性，这是因为只要事务管理器的 id 是 TransactionManager，就可以省略该属性。
