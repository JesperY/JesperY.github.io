HTTP 基础知识

发表于2023-02-24|更新于2023-03-07

1. 什么是 HTTP

HTTP，即超文本传输协议，是一种应用层协议，规定了浏览器和服务器之间传输数据的规则

2. HTTP 基本格式

2.1 请求格式

请求的数据分为三部分：

  1. 请求行：请求数据的第一行，只有三个内容：请求方法、请求路径、协议版本

  2. 请求头：第二行开始为请求头，是一组键值对信息，包含了许多浏览器信息和双方需要约定的信息。

  3. 请求体：POST 请求会在请求体中携带数据，请求体与请求头以空行分隔，请求体传输的数据格式非常丰富。

2.2 响应数据格式

响应数据也非为三部分：

  1. 响应行：分为三部分：协议版本、响应状态码、状态码描述。

  2. 响应头：响应行之后跟随响应头，指明响应数据的相关信息，例如格式、长度，编码等信息。

  3. 响应体：以空行分隔，响应数据存储在响应体中。

3. 会话跟踪技术

HTTP 是无状态的连接，两次不同的请求之间互相独立，数据不共享，但我们经常需要使用之前的请求所传递的信息，因此就需要在服务器或者浏览器将需要的信息存储下来，这就是会话跟踪技术。

3.1 客户端会话跟踪技术

客户端跟踪会话使用 Cookie，Cookie 在请求过程中由服务器创建并响应，浏览器获得 Cookie 将其缓存下来，之后在对同一网站进行访问时会携带该Cookie 信息，服务器解析携带的 Cookie 就可以得到之前请求过程中通过 Cookie 所保存的信息。

Cookie 并不是无期限的存储在浏览器中，而是有着存活时间的限制。

在默认的情况下，Cookie 存储在内存中，这种存储方式随着浏览器的关闭而销毁，我们可以通过配置来改变 Cookie 的存储方式。

Cookie 不能直接存储中文，因此需要转码。

3.2 服务端会话跟踪技术

服务端使用 Session 来跟踪会话。

JavaEE 提供了 HttpSession 接口来规范 Session 的行为。

Session 底层是基于 Cookie的。

设想一下，我们在第一次接收到一个新的浏览器请求时，为其创建了一个 Session 对象并保存在服务器中，当这个浏览器再次发起请求时，我们如何知道这个请求对应哪个 Session 对象呢？毕竟 HTTP 是无状态的，服务器中又存储着很多 Session，因此我们仍然需要根据 Cookie 来识别请求的来源。

服务器创建 Session 会为其设置一个唯一 id，web 服务器会自动将 SessionId 的 Cookie 回传给浏览器，这样浏览器下次访问就可以根据 Cookie 找到对应的 Session。

根据约定，Cookie 中使用 JSESSIONID 作为 key 来存储 SessionId。

Session 的钝化，活化

Session 的数据存储在服务器内存中，当服务器正常关闭时，会将信息存储到硬盘中，这就是 Session 的钝化，服务器重启后会重新加载这些信息并从硬盘上删除，这就是 Session 的活化。

Session 的销毁

Session 默认的销毁策略是 30 分钟未操作就进行销毁，这个值可以通过配置文件修改，也可以手动调用其销毁方法。
