AOP 基础知识

发表于2023-02-10|更新于2023-02-20

1. 什么是 AOP

AOP，Aspect Oriented Programming，面向切面编程。

AOP 是 OOP 的延申，在我们常规的 OOP 思想中，将显示个体抽象为对象，对象有自己的属性和方法，对外提供访问接口，但有时候我们希望对象间协同运作，即 A 对象的某个方法执行完毕后再执行 B 对象的某个方法，或者对 C 对象的方法进行增强使其具备更多的功能，但又不适合改写 C 对象或者继承 C 对象，此时我们可以通过代理模式的思想，在外面包装一个类来对原有的方法进行增强。这种方法就好像把原有的对象一个个切开来，只取用其中的某个切面上的功能。

浅薄认识，仅作了解。

2. 在 Spring 中使用 AOP

我们将 Bean 全部给交给 Spring 管理，使用时也是从 Spring 容器中获取，因此我们可以在 Spring 创建 Bean 的过程中，通过其开放的接口，例如 BeanPostProcessor，创建目标 Bean 的代理对象，通过代理对象增强后，将代理对象加入单例池，如此一来我们在获取目标 Bean 时就会获取到增强后的 Bean。

可以通过实现 Aware 接口在后处理器中获取增强 Bean 所需要的属性。

Spring 已经为我们封装好了 AOP 相关的内容，因此我们并不需要从头编写。

在使用 Spring AOP 之前需要知道的一些术语

术语 单词 解释
目标对象 Target 被增强的目标对象
代理对象 Proxy 增强后实际使用的对象
连接点 Joinpoint 目标对象中可以被增强的方法
切入点 Pointcut 目标对象中实际被增强的方法
通知 / 增强 Advice 增强部分的逻辑，即增强本身
切面 Aspect 增强和切入点的组合，即增强代码与切入点代码结合后的整体成为切面
织入 Weaving 将通知和切入点动态组合的过程，在 Spring 中织入采用的是后处理器
2.1 基于 xml 配置的 AOP

  1. 导入 AOP 包

    Spring 四大核心组件中已经包括了 spring-aop，因此我们只需要导入另外一个 org.aspectj.aspectjweawver 包。

    AspectJ 是一个基于 Java 的 AOP 框架，Spring 2.0 将 AspectJ 纳入 Spring 生态。

  2. 确定目标类、增强类

    将两者提交给 Spring 管理。

    aop:pointcut 标签配置要增强的目标类的目标方法，其 expression 值指定了全限定方法名，即 返回值 全限定类名.方法名()，此处使用的无参方法。

    aop:aspect 标签指定了增强类以及目标方法与增强方法的先后顺序。

  3. 切点表达式

    aop:pointcut 标签，即切点表达式，用于指定要被增强的方法。可以采用上述形式配置多个切点表达式。

    切点表达式也可以写在 aop:aspect 标签内部。

    除了使用 execution 属性外，也有其他的属性可以使用，但使用频率很低，此处暂不讨论。

    execution 的值的写法：

    访问修饰符可以省略不写。

    返回值类型、包名、类名、方法名中，都可以使用 * 来表示任意匹配。

    包名与类名之间使用 . 单点，表示该包下的子包或类，使用 .. 双点表示该包及其子包下的类。

    参数列表可以 .. 双点表示任意参数。

    写法有点类似于简单的正则

    <aop:aspect> 子标签有以下几种。

    通知名 配置方式 执行时机
    前置通知 aop:before 目标方法前执行
    后置通知 aop:after-returning 目标方法后执行，目标方法异常时不执行
    环绕通知 aop:around 目标方法前后均有代码执行，目标方法异常时，环绕后的代码不执行
    异常通知 aop:after-throwing 目标方法异常执行
    最终通知 aop:after 不管目标方法是否异常，始终在目标方法后执行

    aop:around 方法有一个参数，该参数封装了切入点，通过该对象即可调用切入点方法，我们可以在调用前后编写自己的代码，因而实现环绕执行，其他方法为该方法的衍生。

    上述各种通知中，只有 around 环绕通知默认传递一个参数，来调用目标方法，其他通知默认无参。

    增强方法由 Spring 管理调用，参数也由 Spring 传递，我们可以通过配置来让 Spring 传递一些必要的参数。

    1. JoinPoint：连接点对象
    2. ProceedingJoinPoint：正在执行的连接点对象
    3. Throwable：异常对象，需要在标签中配置 throwing="argName" 来指定传入的形参名。
2.2 xml 配置的两种方式

Advisor

Spring 提供了一个 Advice 接口，实现该接口的类都可以作为通知类。

开发中使用的是 Advice 的子接口，其功能各有不同，个别子接口的实现方法也有所区别，使用时请查阅文档。

使用 Advisor 配置时，通知的类型会由其所实现的接口指定，因此不再需要配置诸如 aop:before 之类的标签，而只需要指定要结合的通知类和切点类。

Aspect

此配置即是上节基于 xml 配置 AOP 中所使用的配置方法，此处不再讨论。

Advisor 的通知类通过实现不同接口来执行不同的通知。

Aspect 则使用子标签指定多个通知类型。

Aspect 指定不同通知类型只需要修改标签，而 Advisor 则需要修改或者编写新的通知类，因此 Aspect 适用于需要多种通知类型灵活配置，而 Advisor 适用于通知类型固定的情景。

3. xml 配置 AOP 原理

Spring 解析第三方 xml 标签的原理请查看

Spring 解析 xml 文件

spring-aop 包下注册了 ConfigBeanDefinitionParser 类用于解析 aop:config 标签，其中的 parse() 方法调用了 configureAutoProxyCreator() 配置一个自动代理创建器，其中调用了 AOPNamespace 工具类方法来注册了一个 AspectJ 自动代理创建器。该方法底层判断是否存在一个名为 org.springframework.aop.config.internalAutoProxyCreator 的 BeanDefinition，如果不存在则创建一个，其中保存的Bean 类信息是 org.springframwork.aop.aspectj.autoproxy.AspectJAwareAdvisorAutoProxyCreator 。

AspectJAwareAdvisorAuroProxyCreator 继承自 BeanPostProcessor ，通过后处理器介入 Spring 实例化 Bean 的过程。该后处理器方法底层通过 JdkDynamicAopProxy 对象调用 Proxy.newInstance() 来创建一个代理类。

除了使用 JdkDynamicAopProxy ，还有另外一个实现类 CglibAopProxy 。前者使用 Java 原生提供的 Proxy 类，基于接口生成代理对象，目标类有接口使用该方法。而后者使用 Cglib 动态代理技术，目标类无接口且不能被 final 修饰，通过继承目标对象生成子类作为代理类，目标无接口时采用该方式，或者在 aop:config 标签中配置属性 proxy-target-class="true" 强制使用 Cglib 方式。

4. 注解方式配置 AOP

对于目标类和通知类使用 Bean 注解。

通知类额外使用 @Aspect 注解指定通知类，类内部方法使用 @Before("execution(....)") 等类似注解来指定通知类型和切点表达式。

通知类型的注解名与 xml 通知类型标签同名，只是改用大驼峰命名法。例如 @AfterThrowing("")。另外该 @AfterThrowing 注解需要额外的参数。

@AfterThrowing(pointcut = "xxxx", throwing = "eee")

xml 文件中需要开启 AOP 自动代理。

将切点表达式硬编码在注解中，如果要修改就会很麻烦，需要同时修改多个注解，我们可以通过在通知类中声明一个 myPointcut() 方法，并通过 @Pointcut("") 注解来指定。

使用配置类替代 xml 配置。

5. 注解配置的原理

Spring 解析 AspectJ 注解的原理同解析 xml 配置的原理基本一致，只是在创建名为 org.springframework.aop.config.internalAutoProxyCreator 的 BeanDefitino 时，其内部保存的 Bean 的类信息变成了 AnnotationAwareAspectJAutoProxyCreator 。该类同样继承了 BeanPostProcessor 接口，然后采用与 xml 配置完全相同的方法创建代理对象。

AnnotationAwareAspectJAutoProxyCreator 其实是 AspectJAwareAdvisorAutoProxyCreator 的子类。

@EnableAspectJAutoProxy 注解内部导入了 AspectJAutoProxyRegistrar.Class ，该类通过实现 ImportBeanDefinitionRegistrar 接口，向 Spring 注入 AnnotationAwareAspectJAutoProxyCreator 的 BeanDefinition。

对于上述两节的描述可能存在一些模糊不清的地方，此处再做总结。

xml 解析

Spring 通过 xml 命名空间配置，找到 AspectJ 的标签解析器，该标签解析器通过 parse() 方法创建一个 BeanDefition，其中保存的类信息为AspectJAwareAdvisorAutoProxyCreator 。该类继承了 BeanPostProcessor 接口，从而介入 Bean 的创建过程，将目标对象替换为代理对象。

注解解析

使用 xml 配置开启注解时，Spring 同样使用 xml 标签解析器，只是这次创建的 BeanDefinition 中保存的类信息为 AnnotationAwareAspectJAutoProxyCreator 。

以上两者在 Spring 缓存中的别名相同，对 Spring 隐藏了细节，因此 Spring 对待两者的行为是一样的，只是实际产生的 Bean 对象类型不同，因此行为不同。

使用配置类注解 @EnableAspectJAutoProxy 开启注解配置时，该注解中 @Import(AspectJAutoProxyRegistrar.Class) ，导入了另一个配置类，这个额外的配置类实现了 ImportBeanDefinitionRegistrar 接口，众所周知，被 @Import 导入的类会被自动注入到 Spring 容器，因此 Spring 可以识别到这个接口，从而自动调用该接口的方法，正是通过这个方法向 Spring 中注册了 AnnotationAwareAspectJAutoProxyCreator 的 BeanDefinition。
