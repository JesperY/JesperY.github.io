Spring-mvc 基础知识

发表于2023-02-15|更新于2023-03-07

先来看一下 JavaEE 三层架构和 MVC 三层架构的结合。

JavaEE 三层架构分为表示层，业务层和持久层，对应图中的 web 层，service 层和 dao 层。

而表示层又采用 MVC 三层架构，即 Model，View，Controller。

Jsp 作为 View 用于展示数据并和用户交互，Controller 用于处理请求并调用业务代码，Model 保存数据和进行业务处理。

两种架构结合后，将 Model 中的业务处理合并到 Service 层，Model 专注于数据保存，发展成 POJO。

Tips：spring-mvc 等价于 spring-webmvc，后者是官方名称，前者是常用简称。

1. Spring 与表示层的结合

Spring-web 组件仅仅提供了将 Spring 容器整合到 web 工程中的基本功能，即封装了加载 Spring 配置的代码。

spring-web 通过封装一个 Listener，根据 web.xml 中配置的初始化参数来加载用户的配置文件或者配置类，从而启动 spring 容器。

通过 spring-web 可以使用 Spring 管理业务层和持久层的 JavaBean，而 Servlet 通过 ServletContext 来获取容器。

Servlet 由 web服务器创建和管理，并不是由 Spring 管理，spring-web 通过将容器对象保存到 ServletContext，从而使该 Web 应用中运行的 Servlet 都可以获得 Spring 容器的引用。

想当然的我们会考虑将 Servlet 也交给 Spring 容器管理，从而实现将其他 Bean 直接注入到 Servlet，就像 Service 层的行为一样。但实际上这么做的时候 Spring 并不会为我们实现属性注入。

这是为什么？

先来考虑我们在一个简单的 Java 程序中如何使用 Spring：

  1. 首先，我们通过 Main 函数启动 Java 程序。
  2. Main 函数加载配置文件或者配置类。
  3. Spring 根据配置加载 Bean 并完成依赖注入等一系列操作。
  4. Spring 容器完成创建。

那么 Tomcat 是如何加载 web 工程的呢？

  1. Tomcat 启动，读取 web.xml 中的 context-param 标签，创建 Servlet Context，保存初始化参数。
  2. 按照 Listener，Filter，Servlet 的顺序加载三大组件类（即实现了这三个接口的类）。
  3. 毫无疑问，spring-web 提供的 Listener 将会被首先加载，该类监听 Servlet Context 的创建，因此其 initialization 方法会被调用，开始初始化Spring 容器，此时我们定义的 Bean 都会完成创建。如果我们将一个 Servlet 指定为 Bean，此时也会加载。但是这个 Servlet 对象保存在 Spring 中，而不是 Tomcat 中。
  4. Tomcat 继续加载，直到加载完毕并启动服务。
  5. 由于 Tomcat 在首次接受对应请求时才会加载对应的 Servlet。因此当我们访问时 Tomcat 发现自己并没有加载对应的 Servlet，因此加载一个新的 Servlet，但这个 Servlet 并不是由 Spring 管理的 Servlet，所以其中的依赖也并没有注入，进而导致程序出现错误。

由此可见，这个过程中 Spring 加载的 Servlet 和 Tomcat 加载的 Servlet 并不是同一个，即两个不同的引用。

2. Spring MVC 前端控制器

Spring mvc 框架将 Servlet 的功能做了拆分，提供了一个 DispatcherServlet 作为前端控制器，负责拦截所有请求，并分发给 Controller。

@Controller 是 @Component 的衍生注解，旨在语义化，其功能和 @Component 一样，该注解标注的类会作为 Bean 被 Spring 管理。

实际上我们依然没有将 Servlet 交给 Spring 管理，而是通过 Spring-webmvc 提供的 Servlet 来接受所有的请求，根据路径分发的对应的 Bean 来处理，这个拦截所有请求的 DispatcherServlet 就是表示层与业务层交互的关键。

根据 Spring-web 的基本知识一文，我们可以通过 spring-web 提供的 Listener 来加载配置文件，但当我们使用 spring-webmvc 时，其实际上对 spring-web 做了进一步封装，将配置文件的加载方法封装到了 DispatcherServlet 的 init 方法中，因此我们需要给 DispatcherServlet 提供初始化参数，否则 Tomcat 在加载 DispatcherServlet 时会报错—— init 方法中未找到配置文件。

init 方法默认寻找 /WEB-INF/DispatcherServlet-servlet.xml 文件。

由于 DispatcherServlet 接管了加载配置文件的功能，web 容器只有在第一次访问该 Servlet 时才会创建该 Servlet，而我们想要 Spring 容器在 Tomcat 启动时就加载，因此需要通过 load on startup 指定 DispatcherServlet 在 web 容器启动时加载。

至此 DispatcherServlet 配置完成，Spring-webmvc 接管了传统 Servlet 的工作同时也加载了 Spring 容器。

接下来就是如何使用 DispatcherServlet 分发请求。

5. Spring MVC 组件

Spring MVC 是将 web 和 Spring 通过 MVC 设计模式组合的框架，其中其主要作用的有以下几个组件。

客户端请求首先经过 DispatcherServlet 处理，调用 HandlerMapping 来分析请求路径，构建处理器执行链，将执行链交给 HandlerAdapter，按照执行链进行调用并返回处理结果，DispatcherServlet 根据返回的视图信息调用 ViewResolver 解析，将处理好的视图视图对象返回给客户端。

HandlerMapping、HandlerAdapter、ViewResolver 都是由 Spring MVC 提供的接口规范，我们实际使用的是其实现类。

在 org.springframework.web.servlet 包下有一个 DispatcherServlet.properties，其中指定了每个接口当前所使用的具体实现类，部分接口会指定多个实现类。在 DispatcherServlet 中维护着 HandlerMappings、HandlerAdapters 等集合，集合内部存储着接口对应的实现类对象。DispatcherServlet 的 initStrategies 方法负责根据 ApplicationContex 的信息来初始化各个集合。

DispatcherServlet 在初始化这几个集合之前，会先在 Spring 容器中寻找是否已经存在实现了对应接口的 Bean，如果找到则使用，不再根据 DispatcherServlet.properties 进行初始化。这允许用户通过注入自己的对象来代替 spring-mvc 默认的配置。

6. Spring MVC 如何处理请求

6.1 如何配置请求路径映射

使用注解来配置映射

@RequestMapping 、@GetMapping 、@PostMapping 三者用在方法和类上，可以指定其请求路径，根据名字不难看出他们的功能，当类上和方法上同时指定是，完整路径是类和方法路劲的拼接。

6.2 获得请求数据

通常请求都会跟随很多请求数据，我们要根据请求数据进行处理。

最常见的就是键值对形式的数据。

对于 Get 方法传递的键值对参数，可以通过直接在方法形参中指定键名的形式获取。

Spring MVC 会根据键名匹配，将键值对参数传入方法中，如果参数名不匹配，方法依然会被调用，但只会传入空参数。

如果确实需要使用新的形参名或者因为其他原因形参名不能和键名对应，可以通过注解来指定。

如果提交的参数中，同一个键名对应多个值，可以使用数组来接受

如果想要使用集合对象来接受数据，需要使用注解，该注解仅用于通知 Spring 不要根据形参类型来创建参数对象，因为我们形参通常使用接口作为数据类型，而接口是无法创建对象的。

之所以要告诉 Spring 不要根据形参类型来创建参数对象，是因为后面我们会使用 model 层来接受数据，Spring 会自动根据形参 new 一个对象来封装数据。

同理对于键值对类型，可以使用 Map 直接收集其完整的键值对信息。

@RequestParam 的其他参数属性：

  • required = true 要求该参数必须提供。默认为 false，

    对于引用类型未提供会赋值为 null，但基本数据类型会报错，因此通常使用包装类。

  • defauleValue = "" 指定没有该参数值时使用的默认值

通常形况下，我们会使用 model 层的 pojo 来封装用户数据或者请求参数，在调用业务层时也是将 pojo 直接传递给后续方法，因此如果我们手动将数据封装未免过于繁琐且低效，Spring MVC 提供了自动封装的方法。

localhost:8080/get2?username=test1&age=11&hobbies=bbbb&hobbies=aaa&birthday=2011/2/2&address.city=city&address.area=area

其中 adderss 同样为自定义对象，因此对其参数的设置采用 address.area=area 的形式，即 fieldName.field 的形式，这是约定俗成的写法，Spring 会正确解析。

对于 POST 请求的键值对参数，其本质与 GET 并无不同，只是在请求体中提交，此处不再讨论。

而 POST 还有另外很多常用的数据格式，比如 raw、二进制等，并没有键名，Spring MVC 就无法帮我们自动传入，此时就需要使用注解，指定请求体中的数据全部传入一个形参中。

6.4 处理 Restful 风格的请求参数

Restful 风格参数是一种请求和响应的规范。

Restful 请求遵循以下三点：

  • 使用 URI 表示某个模块资源，该资源名称使用名词指定，请求参数直接以请求路径的形式拼接。

    http://localhost/user 表示对 user 资源的操作。

    http://localhost:/product 表示对 product 资源的操作。

    http://localhost/user/2002 表示对 user 资源操作时传入一个参数，参数值为 100。

    POST 等请求参数在请求体中，不通过 URL 体现。

  • 使用请求方式来指定业务动作。

    例如 GET 方法表示查询，POST 方法表示插入，PUT 方法表示更新， DELETE 方法表示删除。

    http://localhost/user/100 使用 GET 请求时，表示对 user 资源进行查询，传入参数值 100。

  • 使用响应状态码表示结果

    200 表示成功，404 表示未找到，500 表示服务器内部错误。

通过占位符和注解处理 Restful 风格请求。

路径映射表示其接受 /get2/user 的处理，因此 localhost/user/100 将会由该方法处理，其中参数所在的位置被占位符{id}替代，表示这个地方作为参数，其键名为 id ，在方法形参中使用 @PathVariable 注解来指定当前参数匹配哪个占位符，Spring MVC 会根据占位符名自动传入。

如果不使用注解，将无法处理这种风格的请求，因为请求 URL 中只有参数值，没有键名。

6.8 静态资源的访问

web 容器在处理请求时，都是匹配 servlet-mapping，寻找 servlet 来处理请求，即使我们请求一个 .jsp/.html 页面也是如此，但为什么在 Javaweb 中总是能够获取到页面呢？这是因为当对静态资源的请求无法转发到任何一个 servlet 时，会有 web 容器默认提供的 defaultServlet 来处理，这个 servlet 拥有解析静态资源的能力，所以在传统 Javaweb 开发中，实际上是 web 容器帮我们完成了对静态资源请求的处理。

我们使用 Spring MVC 时，会将 DispatcherServlet 的 servlet-mapping 配置为缺省，即拦截所有请求，这种配置覆盖了 web 容器对静态资源的处理，因此我们不能再直接访问静态资源。

一个比较简单的获取静态资源的方法是在工程的 web.xml 中使用目录匹配或者扩展名匹配等高于缺省匹配的配置，来重新激活 defaultServlet。

Spring MVC 提供了两种配置静态资源的方式

配置静态资源映射

使用 xml 配置，将对资源的请求直接映射到资源目录。

这个配置将匹配 /img/* 的所有请求直接映射到工程的 /img/ 目录下，从而直接获取静态资源。

配置默认 servlet

需要注意！

当我们使用 <mvc:default-servlet-handler/> 时，该标签的处理会向 Spring 注入一个 SimpleHandlerMapping。根据第五节关于 Spring MVC 组件的内容，当 Spring 中有了一个 HandlerMapping 的实现类时，DispatcherServlet 将不会再加载默认的 HandlerMapping，Spring MVC 在其配置文件中提供了三个默认的 HandlerMapping，其中名为 RequestMappingHandlerMapping 的类负责处理 @RequestMapping 注解，此时这个类不会被加载，也就意味着 Spring 失去了解析 @RequestMapping 的能力，对 Controller 的访问将不可用，因此在使用该标签时，我们需要手动注入 RequestMappingHandlerMapping。

使用 <mvc:resources> 标签时同理。

bean 的配置必须在前面，否则不生效。

7.1 同步数据

请求资源转发

请求转发，即服务器内部转发，在传统 Javaweb 使用 dispatcher.forward() 实现，而 Spring MVC 接管了 Servlet 的处理，因此我们只需要在 Controller 方法的最后返回一个字符串来表示要转发的路径即可。

这里返回的字符串省略了一些内容，请求转发的完整字符串实际是 forward:/request2 ，前缀指定进行请求转发，省略时默认使用 forward。

请求重定向

重定向在传统 Javaweb 中使用 resp.senddirect() 的形式完成，在 Spring MVC 中只需要将 forward 换成 redirect 即可。

重定向是由浏览器重新发起一次请求，因此不能共享 request 域数据。

响应模型数据

MVC 架构中出了我们一直在讨论的 Controller，还有 Model 和 View，前者用于存储数据，后者用于展示数据，在前面我们曾试过使用 Model 直接接受来自浏览器的数据，这就是 Model 的作用。而对浏览器请求时就要使用到 View，通常 Model 和 View 相互绑定，前者提供要处理的数据，后者对数据进行展示。

Spring MVC 为我们提供了一个 ModelAndView 对象，该对象中封装了模型数据 Model 和视图名。

我们通过在形参中指定 ModelAndView 来让 Spring 传入一个模型视图对象，然后将要传递给视图的数据封装在对象中，并设置进行展示的视图名，Spring MVC 会将这些数据传递给视图。

获得 ModelAndView 的方式很多，如果希望使用形参获取请求参数，则可以在方法体中自己 new 一个 ModelAndView。

ModelAndView 的处理方式依赖于 jsp，基本已经废弃。

直接写回数据

可以自己直接写回字符串数据，由于 Spring MVC 默认将返回的字符串识别为请求转发，因此需要通知 Spring MVC 我们不进行转发，直接响应。

7.2 异步数据

异步数据通常写回给 Ajax 引擎，通常都是 Json 字符串。

自己直接创建一个 Json 字符串是比较繁琐的，而我们的数据又保存在 pojo 中，pojo 存储的信息可以比较方便的转换为 Json 格式，因此我们借助一些工具来进行转换。和接受 Json 字符串时一样，Spring MVC 提供了整合的功能。只要使用了 @ResponseBody 注解，我们可以直接返回一个 pojo 对象，Spring MVC 会自动将其转换为 Json 字符串。

目前的开发方式通常都是 Restful 和 异步 Json 结合的方式，即请求使用 Restful，响应使用 Json，一个 Controller 中的方法都是用 @ResponseBody 注解，这种情况下就可以把 @ResponseBody 和 @Controller 注解合并为 @RestController，类上使用这个注解即可。

使用了 @RestController 相当于在所有方法上使用了 @Response 注解，此时将不再进行重定向和请求转发，请务必注意。

8. Spring MVC 拦截器

拦截器功能类似于过滤器，但是过滤器在 Servlet 之前生效，而拦截器在 DispatcherServlet 之后，Controller 之前生效。

HandlerInterceptor 接口声明了三个方法

参数由 SpringMVC 传递，handler 顾名思义是处理器，也就是最终调用的 Controller 的相关信息。

配置多个拦截器时会按照配置的顺序先后执行。就像一个执行链，第一个拦截器的 pre 后是第二个拦截器的 pre，但是 Handler 完成后会先执行 第二个 post，然后是第一个的 post。可以理解为拦截器之间是调用链的方式，其行为像栈。

任何一个拦截器的 pre 如果返回 false，Handler 就不会执行，进而导致所有拦截器的 post 都不执行，但是 after 的执行与否只取决于其拦截器本身的 pre，因此所有返回 true 的拦截器的 after 都会执行。

拦截器的原理

SpringMVC 使用 RequestMappingHandlerMapping 来寻找对应的 RequestMapping，当我们配置拦截器时， RequestMappingHandlerMapping 会首先去匹配拦截器，然后再寻找 RequestMapping，将拦截器和最终的 Controller 构建一个 ExecutionChain 返回给 HandlerAdapter 处理。

9. SpringMVC 的注解配置

此部分只设计 SpringMVC 组件的注解，Spring 等注解不再讨论。

SpringMVC xml 文件中主要配置了 mvc:annotation-driven、mvc:default-servlet-handler 和 mvc:interceptor 三个标签。这三个标签可以使用一个注解 @EnableWebMvc 完成配置，该注解导入了 DelegatingWebMvcConfiguration 类，该类及其父类会完成注册 HandlerMapping、HandlerAdapter。除此之外我们要实现一个 WebMvcConfigurer 接口，该接口中的方法将会完成注册拦截器、配置 DefaultServletHandler。SpringMVC 会自动从 Spring 容器中获取该接口的实现并调用其方法。

首先我们编写一个配置类

然后我们要继承一个 AnnotationConfigWebApplicationContext 来加载我们自己写的配置类。这是因为 DispatcherServlet 要求提供这样一个子类，以便它可以调用其中的方法，这个子类才是 MVC 对外暴露的接口。

如果我们想要使用拦截器和 DefaultServlet，还需要实现 WebMvcConfigurer 接口，并将其实现类注入 Spring 容器。

至此我们对于配置文件中的所有配置项都已经完成了注解配置，接下来需要修改 DispatcherServlet 的初始化参数加载配置类。
